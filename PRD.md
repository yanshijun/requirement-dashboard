## 一、项目概述

### 1.1 项目简介

需求池看板是一个面向产品研发团队的需求管理工具，用于统一管理客服GPT SAE版产品的需求全生命周期。支持需求录入、状态跟踪、人员分配、开发进度看板、日常反馈收集等功能。

#### 1.2.2 技术栈明细

| 层级 | 技术 | 说明 |
|------|------|------|

#### 1.2.5 后端架构设计

**双模式运行**：同一份核心逻辑（`feishu.js`）可在两种环境运行。

```
模式一：Express 独立服务（server.js）
┌──────────────┐      ┌─────────────────────┐      ┌──────────┐
│   浏览器      │─────→│  Express Server      │─────→│ 飞书 API  │
│              │      │  · 路由 /api/feishu   │      │          │
│              │←─────│  · 静态文件托管        │←─────│          │
│              │      │  · 调用 handler()     │      │          │
└──────────────┘      └─────────────────────┘      └──────────┘

模式二：Netlify Serverless
┌──────────────┐      ┌─────────────────────┐      ┌──────────┐
│   浏览器      │─────→│  Netlify CDN         │─────→│ 飞书 API  │
│              │      │  · 静态文件分发        │      │          │
│              │←─────│  · Functions 路由     │←─────│          │
│              │      │  · feishu.js handler  │      │          │
└──────────────┘      └─────────────────────┘      └──────────┘
```

#### 1.2.6 数据流设计

**读取数据流**：

```
页面加载 → loadData()
              │
              ├─ GET /api/feishu?table=all
              │       │
              │       ▼
              │  handler() → getToken() → 并行拉取
              │                           ├─ getAllRecords(需求表)
              │                           └─ getAllRecords(反馈表)
              │       │
              │       ▼
              │  mapFromFeishu() / mapFromFeedback()
              │       │
              ▼       ▼
         allData[] = res.requirements
         feedbackData[] = res.feedback
              │
              ▼
         buildPersonFilter() → render()
```

**写入数据流（乐观更新）**：

```
用户操作（保存/删除/认领）
       │
       ├─① 立即更新本地 allData[]
       ├─② 立即调用 render() 刷新视图
       ├─③ 关闭弹窗
       │
       └─④ 异步 POST /api/feishu?action=xxx
              │
              ├─ 成功 → loadData(true) 后台静默同步真实数据
              └─ 失败 → alert 错误信息 → loadData() 重新加载
```

#### 1.2.7 飞书认证流程

```
getToken()
    │
    ├─ 缓存有效？（_tokenCache && Date.now() < _tokenExpiry）
    │   ├─ 是 → 直接返回缓存 token
    │   └─ 否 ↓
    │
    ├─ POST /auth/v3/tenant_access_token/internal
    │   body: { app_id, app_secret }
    │       │
    │       ▼
    │   _tokenCache = data.tenant_access_token
    │   _tokenExpiry = Date.now() + 90min
    │       │
    └───────▼
       返回 token
```

### 2.6 新增/编辑需求

#### 2.6.1 表单字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 需求编号 | 文本（只读） | 自动 | 格式 URQ-N（主需求）或 URQ-N-M（子需求），自动生成，不可编辑 |
| 需求类型 | 下拉 | * | 用户需求 / 研发需求 |
| 需求名称 | 文本 | * | 必填 |
| 需求描述 | 多行文本 | 否 | 支持 Markdown，可上传 .md 文件，支持编辑/预览切换 |
| 所属计划 | 下拉 | 否 | 支持动态新增和删除计划 |

| 预计上线日期 | 日期 | 否 | - |
| 提出时间 | 日期 | 否 | 新增时默认当前日期 |
| 是否需要设计 | 下拉 | 否 | 待设计（默认）/ 需要设计 / 不需要设计 / 设计完成 |
| 评审是否通过 | 下拉 | 否 | 待确认（默认）/ 已通过 / 已拒绝 / 暂缓 |
| 备注 | 多行文本 | 否 | - |

#### 2.6.2 Markdown 描述

- 支持 Markdown 语法编写需求描述
- 编辑/预览 Tab 切换
- 支持上传 .md / .txt 文件导入
- 预览使用 marked.js 渲染

#### 2.6.3 子需求管理

- **新增子需求**：在需求详情或列表中点击"+子"按钮快速添加子需求
- **编号自动生成**：子需求编号格式为 URQ-N-M（N 是父需求编号，M 是子需求索引）
- **认领人同步**：子需求认领人会自动同步到父需求（合并，不覆盖）
- **进度聚合**：父需求进度自动计算为所有子需求的平均进度

#### 2.6.4 计划管理

- **默认计划**：SAE2.3.3、1688商贸版（不可删除）
- **新增计划**：点击"+ 新增"按钮，输入名称后添加到下拉列表
- **删除计划**：点击"删除"按钮移除当前选中的自定义计划
- 自定义计划存储在 localStorage（`extraPlans`）

#### 2.6.5 保存行为

- **乐观更新**：保存后立即更新本地数据并刷新视图，后台静默同步飞书
- **新增**：调用 `api('add', item)`，成功后将新记录加入 allData
- **编辑**：调用 `api('update', item)`，成功后更新 allData 对应记录

### 2.9 数据导入

#### 2.9.1 导入流程

1. 打开飞书多维表格 → 全选数据行 → Ctrl+C 复制
2. 粘贴到导入弹窗的文本框
3. 系统自动解析并预览数据
4. 确认无误后点击"确认导入"

#### 2.9.2 列顺序要求

```
需求编号 · 需求名称 · 需求描述 · 需求类型 · 所属计划 · 优先级 · 开发状态 · 开发进度 · 需求认领人 · 预计上线日期 · 是否需要设计 · 评审是否通过 · 备注 · 父需求ID
```

#### 2.9.3 导入行为

- Tab 分隔符解析
- 预览最多显示前20条
- 名称为空的行自动跳过并提示
- **覆盖导入**：先删除飞书表中所有旧记录，再批量创建新记录（每500条一批）

### 2.10 编号管理

#### 2.10.1 自动生成编号

- 新增需求时自动生成编号，格式：`URQ-N`

- 编号字段只读，不可手动修改

#### 2.10.2 重新编号

- 点击顶部"重新编号"按钮
- 按创建时间升序排列
- 依次分配 URQ-1、URQ-2、URQ-3...
- 空编号自动补上，重复编号自动修正
- 逐条调用飞书 API 更新

## 三、API 接口

### 3.1 统一入口

```
GET  /api/feishu?table=all      → 获取需求+反馈全量数据
GET  /api/feishu?table=feedback  → 仅获取反馈数据
GET  /api/feishu                 → 仅获取需求数据
POST /api/feishu?action=add      → 新增需求
POST /api/feishu?action=update   → 更新需求
POST /api/feishu?action=delete   → 删除需求
POST /api/feishu?action=import   → 批量导入需求（覆盖）
POST /api/feishu?action=fb-add   → 新增反馈
POST /api/feishu?action=fb-update→ 更新反馈
POST /api/feishu?action=fb-delete→ 删除反馈
```

## 五、交互规范

### 5.1 乐观更新

所有写操作（新增、编辑、删除、认领、释放）采用乐观更新策略：
1. 立即更新本地 allData 数组
2. 立即刷新视图
3. 后台异步调用飞书 API
4. 失败时提示错误并重新加载数据

### 5.2 Tab 切换

- 切换到"今日收集"时隐藏筛选工具栏
- 切换回其他 Tab 时恢复工具栏
- 切换到"需求列表"时自动重置所有筛选条件
- 今日收集加载中切换到其他 Tab 时，接口返回后不会覆盖当前视图（`curTab` 守卫）

### 5.3 弹窗

- 点击遮罩层关闭弹窗
- 所有弹窗居中显示，最大高度90vh可滚动
- 保存按钮禁用态防止重复提交
