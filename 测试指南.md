## 一句话先搞懂

- **手动测试**：你自己打开浏览器，点来点去，看功能对不对。费时、容易漏。
- **自动化测试**：写一段代码，让程序替你打开浏览器、自动点来点去、自动判断对不对。跑一次几十秒，可重复。

本项目用了两套自动化方案：

| 工具 | 一句话作用 | 谁来写/触发 |
|------|-----------|------------|
| **Playwright** | 把测试步骤写成固定脚本，反复跑、出报告 | 提前写好 `.spec.js` 脚本文件 |
| **MCP（Playwright MCP）** | 让 AI（Claude）实时操控浏览器帮你测 | 你用自然语言让 AI 当场操作 |

## 两个工具分别是什么、有什么用

### 1. Playwright 是什么

**Playwright** 是微软开源的浏览器自动化测试框架。

**核心能力**：用代码控制浏览器，模拟真人操作。
- 打开网页、点击按钮、填表单、选下拉框、上传文件
- 检查页面上有没有出现预期的文字 / 元素（这一步叫"断言"）
- 自动截图、录像
- 同时支持 Chrome、Firefox、Safari 三种浏览器

**在本项目里的角色**：
我们把"创新项目"等模块的测试步骤写成了脚本文件（放在 `tests/` 文件夹）。
每次改完代码，跑一条命令，Playwright 就自动把这些步骤全跑一遍，告诉你哪些通过、哪些挂了。

**典型脚本长这样**（节选自本项目 `tests/bugs.spec.js`）：

```javascript
test('点击行打开详情抽屉', async ({ page }) => {
  // 找到表格第一行，点它
  const firstRow = page.locator('#bugTableArea tbody tr').first();
  await firstRow.click();
  // 断言：详情抽屉应该弹出来了
  await expect(page.locator('#bugDrawer.open')).toBeVisible();
  // 断言：抽屉里应该有"问题描述"这几个字
  await expect(page.locator('#bugDrawerBody')).toContainText('问题描述');
});
```

读法：找到第一行 → 点击 → 检查抽屉弹出 → 检查抽屉内容。
跑的时候这一串动作几秒内自动完成，全对就标 ✓，有一步不对就标 ✗ 并截图。

---

### 2. MCP（Playwright MCP）是什么

**MCP** 全称 Model Context Protocol（模型上下文协议），是 Anthropic 推出的开放标准，作用是**让 AI 连接外部工具**。

打个比方：MCP 就像 USB 接口。电脑有了 USB 口，就能插鼠标、键盘、U 盘。
AI 接了 MCP，就能"插上"各种工具——数据库、文件系统、浏览器等。

**Playwright MCP** 就是其中一个工具：它把 Playwright 的浏览器操控能力，通过 MCP 接口暴露给 AI。

**在本项目里的角色**：
配好之后，你直接用中文跟 AI（Claude）说"帮我打开看板，登录，点进创新项目看看"，
AI 就会实时操控一个浏览器去做这些事，每一步还能截图给你看。
**不需要提前写脚本**，想测什么当场说。

**和 Playwright 脚本的关系**：
MCP 底层用的也是 Playwright 引擎，区别在于"谁来指挥"——
脚本是提前写死的固定流程，MCP 是 AI 当场临时操作。

---

## 它们的区别和配合关系

| 对比项 | Playwright（脚本） | MCP（AI 实时操控） |
|--------|-------------------|-------------------|
| **谁来操作** | 提前写好的脚本文件 | AI 当场理解你的话去操作 |
| **要不要写代码** | 要（写 `.spec.js`） | 不要，说人话就行 |
| **适合场景** | 固定流程反复回归测试 | 探索性测试、临时验证、写脚本前先试 |
| **能不能重复跑** | 能，一条命令跑全部 | 每次都要重新说 |
| **结果形式** | 报告、录像、✓/✗ 列表 | AI 截图 + 文字反馈 |
| **底层引擎** | Playwright | Playwright（同一个引擎） |

**最佳配合方式**（本项目就是这么用的）：
1. 先用 **MCP** 让 AI 当场点一遍，确认功能是通的、找准页面元素。
2. 再把验证过的流程，用 **Playwright** 写成固定脚本，以后每次改代码自动回归。

## ⭐ 日常自动化测试：从零到跑完的每一步

> 这是你以后每次想跑测试时照抄的流程。分两条线：
> **A 线 = Playwright 脚本测试**（自己跑，最常用）；
> **B 线 = MCP 让 AI 帮测**（让我帮你当场操作）。
> 两条线都必须先做"通用准备"。

### 通用准备（两条线都要）

**步骤 1：打开项目所在终端**
确保终端当前目录在 `d:\dashboard`（提示符里能看到 `dashboard`）。
如果不在，先切过去：
```bash
cd d:/dashboard
```

**步骤 2：启动后端服务（关键，不启动测试必失败）**
```bash
node server.js
```
看到 `✅ 服务已启动: http://localhost:3000` 就成功。
**这个终端窗口从此不要关、不要按 Ctrl+C**，让它一直开着。

> 原理：测试和 AI 都要去访问 `http://localhost:3000` 这个网址，
> 这个网址由 `server.js` 提供。服务没起，页面打不开，测试全挂。

**步骤 3：确认服务真的起来了（可选但建议）**
另开一个浏览器，访问 http://localhost:3000 ，
能看到登录页（📋 需求池看板）就说明服务正常。

---

### A 线：用 Playwright 脚本自己跑测试

> 适合：日常回归（改完代码确认没改坏）、想要可重复的结果。

**A-1：新开一个终端**
保留上面跑 `node server.js` 的终端，**另开一个新终端**，同样进到 `d:\dashboard`。

> 为什么要新开？因为第一个终端被 `server.js` 占着（它在持续运行），
> 你需要一个空闲终端来敲测试命令。

**A-2：选一个跑法敲命令**

| 你的目的 | 敲这条命令 |
|---------|-----------|
| 快速跑创新项目全部用例 | `npx playwright test bugs` |
| 跑所有模块的全部用例 | `npx playwright test` |
| 只跑某一个用例（按名字） | `npx playwright test -g "复制按钮"` |
| 想亲眼看浏览器自动操作（Git Bash） | `HEADED=1 npx playwright test bugs` |
| 想亲眼看浏览器自动操作（PowerShell） | `$env:HEADED=1; npx playwright test bugs` |

**A-3：等待并读结果**
终端会一条条打印：
```
  ok  1 tests\bugs.spec.js:25:3 › 创新项目 › 页面加载，显示统计卡片和表格 (4.4s)
  ...
  10 passed (50.3s)
```
- 全是 `ok` / 最后 `X passed` = 全过 ✅
- 出现 `failed` = 有失败，失败用例会自动截图存到 `test-results/` 文件夹

**A-4：失败了想看细节**
```bash
npx playwright show-report
```
浏览器打开一个网页报告，点失败的用例能看到每一步 + 出错截图。

**A-5：跑完收尾**
测试跑完终端会自动回到可输入状态，A 线就结束了。
`server.js` 那个终端可以继续留着，下次还能直接用。

### B 线：用 MCP 让 AI（Claude）帮你测

> 适合：临时验证某个功能、还没写脚本想先试、想让 AI 边操作边截图给你看。

**B-1：确认 MCP 已连接**
本项目根目录已有 `.mcp.json`（配置好了 playwright 这个 MCP 服务器）。

## Playwright 测试：完整操作流程

### 第 0 步：理解项目结构

本项目和测试相关的文件：
```
d:\dashboard\
├── server.js                  ← 后端服务（测试前必须先启动它）
├── index.html                 ← 整个前端页面
├── playwright.config.js       ← Playwright 配置（端口、是否显示浏览器等）
├── .mcp.json                  ← MCP 配置（给 AI 用的浏览器工具）
└── tests\                     ← 所有测试脚本放这里
    ├── bugs.spec.js           ← 创新项目模块测试
    ├── list.spec.js           ← 需求列表测试
    ├── add-edit.spec.js       ← 新增/编辑需求测试
    ├── kanban.spec.js         ← 开发看板测试
    ├── person.spec.js         ← 人员任务测试
    ├── feedback.spec.js       ← 今日收集测试
    └── claim.spec.js          ← 认领测试
```

### 配置文件讲解（`playwright.config.js`）

本项目的关键配置：
```javascript
module.exports = defineConfig({
  testDir: './tests',                          // 测试脚本都在 tests 文件夹
  timeout: 30000,                              // 单个用例最多跑 30 秒
  use: {
    baseURL: 'http://localhost:3000',          // 测试访问的网址（对应 server.js）
    headless: process.env.HEADED ? false : true, // 有 HEADED=1 就显示浏览器
    launchOptions: {
      slowMo: process.env.HEADED ? 600 : 0,    // 有头模式每步放慢 600 毫秒
    },
    viewport: { width: 1440, height: 900 },    // 浏览器窗口大小
    video: process.env.HEADED ? 'on' : 'off',  // 有头模式自动录像
  },
});
```
重点：加不加 `HEADED=1` 决定了你**看不看得到浏览器窗口、有没有录像**。

## MCP 测试：完整操作流程

### 第 0 步：理解 MCP 怎么配的

本项目根目录 `.mcp.json`：
```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp", "--viewport-size", "1440,900"]
    }
  }
}
```
读法：
- `playwright` 是这个 MCP 服务器的名字
- `command` + `args` 是启动命令：`npx @playwright/mcp --viewport-size 1440,900`
- `--viewport-size 1440,900` 指定浏览器窗口大小 1440×900

> 常用参数补充：
> - `--headless`：无头模式，浏览器后台跑、不显示窗口（快，但看不到画面）
> - 去掉 `--headless`：有头模式，显示浏览器窗口

### 第 1 步：让 MCP 生效

`.mcp.json` 是项目级配置。**改完或新建后，需要重启 Claude Code 会话**，AI 才会加载这个工具。
重启后 AI 这边会显示 `playwright` MCP 服务器已连接。

### 第 2 步：先启动开发服务器（和 Playwright 一样）

```bash
node server.js
```

### 第 3 步：用自然语言让 AI 测试

直接说人话即可，例如：
- "帮我打开看板，用 admin / admin123 登录，进创新项目看看"
- "点一下第一行 Bug，看详情抽屉弹出来对不对，截图给我"
- "勾选两条记录，看导出按钮有没有变成'导出选中(2)'"

AI 会依次调用 MCP 工具完成操作。本项目实测时 AI 用到的典型工具：

| 工具 | 作用 |
|------|------|
| `browser_navigate` | 打开指定网址 |
| `browser_snapshot` | 抓取页面结构（比截图更适合 AI 判断元素） |
| `browser_fill_form` | 批量填表单（如用户名、密码） |
| `browser_click` | 点击某个元素 |
| `browser_take_screenshot` | 截图（给人看效果） |

### 第 4 步：看 AI 反馈

AI 把每一步结果、截图、判断直接发在对话里。
比如"详情抽屉正常弹出，标题显示 No.99，附件可点击"，并附上截图。

## 给新功能补测试用例

以后你加了新功能，想让它也进自动化测试，步骤：

**1. 找到或新建对应的 spec 文件**
比如创新项目相关的就加到 `tests/bugs.spec.js`，新模块就新建 `tests/xxx.spec.js`。

**2. 照着已有用例的格式写**
一个用例就是一个 `test('用例名', async ({ page }) => { ... })`，里面三件事：
```javascript
test('我的新功能正常工作', async ({ page }) => {
  // ① 操作：找元素、点击、填值
  await page.click('button:has-text("某按钮")');
  // ② 断言：检查结果符合预期
  await expect(page.locator('#某区域')).toContainText('预期文字');
});
```

**3. 注意登录**
本项目要登录才能进功能页。`tests/bugs.spec.js` 的 `beforeEach` 里已有登录逻辑，
新写的 spec 可以照抄那段（`page.goto('/')` → 填 admin/admin123 → 点登录 → 切 tab）。

**4. 选准选择器（locator）**
- 优先用 id：`#bugTableArea`、`#bugCheckAll`（最稳）
- 按文字找：`text=🐛 创新项目`、`button:has-text("复制")`
- 选择器太宽会"匹配到多个元素"报错，就限定到容器内：`#bugTableArea th:has-text("编号")`

**5. 跑一遍验证**
```bash
npx playwright test 你的文件名
```
绿了就算接进自动化了，以后每次 `npx playwright test` 都会带上它。

> 也可以直接让我（AI）帮你写：先用 MCP 当场点一遍确认元素，再生成 spec。

## 常见问题排查

| 现象 | 原因 | 解决 |
|------|------|------|
| 页面空白 / 测试全挂 | 没先启动 `server.js` | 先在另一个终端跑 `node server.js` |
| 报"浏览器找不到" | 没下载 Playwright 浏览器内核 | 跑 `npx playwright install` |
| `HEADED=1` 没弹窗口 | 用的是 PowerShell，语法不同 | 改用 `$env:HEADED=1; npx playwright test bugs` |
| 看不到浏览器窗口 | 窗口弹在了别的窗口后面 | 看任务栏找新开的 Chrome，或用录像/报告 |
| UI 面板白屏 | 面板渲染兼容问题 | 改用录像或 `show-report` |
| MCP 工具调不出来 | `.mcp.json` 改了没重启 | 重启 Claude Code 会话 |
| 测试报"找到多个元素" | 选择器太宽泛匹配到多处 | 把选择器限定到具体容器，如 `#bugTableArea th:has-text("编号")` |
| 登录卡住 | 早期脚本没写登录步骤 | 在 `beforeEach` 里加登录逻辑（参考 `bugs.spec.js`） |
| `cd` 进不去目录 | 路径写错 | 确认是 `d:/dashboard`，注意正斜杠 |
| 端口被占用 | 上次的 `server.js` 没关 | 关掉旧终端，或换端口 `PORT=3001 node server.js` |

## 名词表（小白速查）

| 名词 | 通俗解释 |
|------|---------|
| **Playwright** | 微软的浏览器自动化框架，用代码控制浏览器 |
| **MCP** | 让 AI 连接外部工具的"USB 接口"标准 |
| **Playwright MCP** | 把浏览器操控能力暴露给 AI 的 MCP 工具 |
| **spec / 用例** | 一个测试场景，对应一段测试代码（如"点行打开抽屉"） |
| **断言 (assert/expect)** | 检查"实际结果是否符合预期"的那一步 |
| **headless / 无头** | 浏览器在后台跑、不显示窗口 |
| **headed / 有头** | 显示真实浏览器窗口，能看到操作 |
| **选择器 (locator)** | 在页面上"定位某个元素"的写法，如 `#bugTableArea tbody tr` |
| **回归测试** | 改完代码后，把以前的测试再跑一遍，确认没改坏旧功能 |
| **slowMo** | 放慢每步操作的间隔（毫秒），方便肉眼观察 |
| **终端 / terminal** | 敲命令的黑框框（VSCode 里 `Ctrl+\`` 打开） |
