# 售后知识库 AI 问答系统需求设计

**文档版本**: v1.0  
**创建日期**: 2026-04-27  
**需求类型**: 新功能设计  

---

## 一、需求背景与目标

### 1.1 背景

当前问题记录模块（客户问题记录表）已积累大量真实售后案例，包含：客户 ID、套餐类型、问题描述、处理状态、责任人、处理附件等结构化信息。这些数据是宝贵的历史经验沉淀，但目前只能手动翻查，效率极低。

### 1.2 目标

将问题记录表作为售后知识库的数据源，结合 LLM（大语言模型）构建 AI 问答能力：

- 输入：客服/售后人员描述的问题场景
- 输出：基于历史案例的处理方案、话术建议、注意事项

### 1.3 价值

| 角色 | 收益 |
|------|------|
| 新人客服 | 快速上手，减少培训依赖 |
| 老客服 | 复杂问题快速检索历史方案 |
| 管理层 | 知识沉淀自动化，减少重复劳动 |

## 二、整体方案设计

### 2.1 架构概览

```
问题记录表（飞书多维表格）
        ↓ 定期全量/增量拉取
  知识条目预处理层（结构化 → 文本化）
        ↓
  [可选] 向量化存储（Embedding）
        ↓
  用户输入问题
        ↓
  召回相关历史案例（关键词 or 向量检索）
        ↓
  组装 Prompt → 调用 LLM API
        ↓
  返回回复方案 + 引用来源案例
```

### 2.2 两种技术路线对比

| 方案 | 描述 | 优点 | 缺点 | 推荐场景 |
|------|------|------|------|----------|
| **方案 A：全量投喂** | 每次查询时将所有历史记录拼入 Prompt | 实现简单，无需向量数据库 | Token 消耗大，记录多时超出上下文窗口 | 记录 < 200 条 |
| **方案 B：向量检索 + RAG** | 先将问题向量化，检索 Top-K 相似案例，再投喂 LLM | Token 可控，可扩展，精准度高 | 需引入向量数据库或 Embedding API | 记录 > 200 条，长期使用 |

**当前阶段建议**：先用方案 A 快速验证，问题记录超过 300 条后升级为方案 B。

## 三、功能详细设计

### 3.1 知识库管理（后台）

#### 3.1.1 知识条目入库规则

并非所有问题记录都适合作为知识库条目，需满足以下条件：

- `处理状态` 为 **已解决** 或 **已关闭**
- `问题描述` 字段不为空（字数 ≥ 10）
- 可手动标记「排除出知识库」（某些记录含客户隐私或特殊情况不适合引用）

#### 3.1.2 知识条目字段设计

飞书表格建议新增字段（或在前端处理时提取）：

| 字段 | 来源 | 说明 |
|------|------|------|
| 知识摘要 | 人工填写 / AI 自动生成 | 对问题+方案的精炼描述，100字以内 |
| 解决方案 | 人工填写 | 实际采用的处理步骤/话术 |
| 问题标签 | 人工打标 / AI 提取 | 如：退款/功能异常/账号问题/套餐升级 |
| 是否入库 | 勾选框 | 默认已解决自动入库，可手动排除 |

#### 3.1.3 知识库同步机制

- **全量同步**：点击「同步知识库」按钮，拉取所有符合条件的已解决记录
- **增量同步**：每次新记录状态变为「已解决」时，自动标记待入库
- 同步后显示：知识库条目总数、最近同步时间

---

### 3.2 AI 问答功能（前端）

#### 3.2.1 入口

在「问题记录」Tab 内，顶部新增「🤖 AI 问答」按钮，点击展开问答面板（不跳转页面）。

#### 3.2.2 问答界面

```
┌─────────────────────────────────────────────┐
│  🤖 售后 AI 助手                          ✕ │
├─────────────────────────────────────────────┤
│                                             │
│  [对话记录区，可滚动]                       │
│                                             │
│  ─────────────────────────────────────────  │
│  📝 请描述遇到的问题...          [发送]     │
│  ○ 仅检索历史案例   ● AI 生成回复方案       │
└─────────────────────────────────────────────┘
```

#### 3.2.3 响应内容结构

AI 回复包含三部分：

```
【AI 回复方案】
根据历史案例，该类问题建议处理步骤为：
1. ...
2. ...
3. ...

【参考话术】
"您好，关于您反馈的XXX问题..."

【引用案例】（可折叠）
• 2026-03-15 | 客户 C001 | 套餐类型：团队版
  问题：xxx   处理方案：xxx
• 2026-02-20 | 客户 C008 | ...
```

#### 3.2.4 模式说明

| 模式 | 说明 |
|------|------|
| 仅检索历史案例 | 不调用 LLM，直接返回最相似的 Top 5 历史案例 |
| AI 生成回复方案 | 检索 + LLM 生成，输出结构化处理方案 |

## 四、Prompt 设计

### 4.1 系统提示词（System Prompt）

```
你是一名专业的客服售后助手，专门处理 SaaS 产品的客户问题。
你的任务是根据提供的历史处理案例，为当前问题给出：
1. 清晰的处理步骤（3-5步）
2. 可直接使用的客户回复话术
3. 需要注意的风险点（如有）

要求：
- 回复专业、简洁，不要废话
- 处理步骤要可操作，不要泛泛而谈
- 如果历史案例不足以支撑回答，请明确说明"历史案例参考有限，建议人工处理"
- 不要编造不存在于历史案例中的信息
```

### 4.2 用户 Prompt 模板

```
【当前问题】
${userInput}

【相关历史案例】
${topKCases.map((c, i) => `
案例${i+1}：
- 日期：${c.date}
- 套餐类型：${c.packageType}
- 问题描述：${c.desc}
- 处理结果：${c.solution || c.status}
`).join('')}

请根据以上历史案例，给出处理当前问题的方案。
```

## 五、接口设计

### 5.1 新增后端接口

#### `POST /api/feishu?action=kb-search`

**功能**：从已解决问题中检索相关案例

**请求 Body**：
```json
{
  "query": "客户无法登录，提示账号已过期",
  "topK": 5
}
```

**返回**：
```json
{
  "cases": [
    {
      "id": "recXXX",
      "date": "2026-03-15",
      "customerId": "C001",
      "packageType": "团队版",
      "desc": "账号登录提示已过期...",
      "status": "已解决",
      "solution": "引导客户续费，临时授权48小时"
    }
  ]
}
```

**检索逻辑（方案 A）**：
- 对 `desc` 字段做关键词分词匹配（简单版：split + includes）
- 按匹配词数排序，取 Top K

#### `POST /api/kb-ai`（新函数）

**功能**：组装 Prompt，调用 LLM，返回 AI 回复

**请求 Body**：
```json
{
  "query": "客户无法登录，提示账号已过期",
  "cases": [...],
  "model": "claude-3-5-sonnet"
}
```

**返回**：
```json
{
  "answer": "处理步骤：\n1. ...\n\n参考话术：...",
  "usedCases": 3,
  "inputTokens": 1200,
  "outputTokens": 350
}
```

## 六、数据安全与隐私

| 风险点 | 处理方式 |
|--------|---------|
| 客户 ID 等敏感信息发送给 LLM | Prompt 中隐去完整客户 ID，仅保留套餐类型、问题类型等非敏感字段 |
| 历史案例包含内部处理细节 | 管理员可标记「不入库」，排除敏感案例 |
| API Key 泄露 | LLM API Key 通过环境变量注入，不在前端暴露 |

## 七、分阶段实施计划

### Phase 1（1-2周）：关键词检索 + AI 问答 MVP

- [ ] 飞书表格新增「解决方案」「问题标签」「是否入库」字段
- [ ] 后端实现 `kb-search` 关键词检索接口
- [ ] 后端实现 `kb-ai` 接口（对接 Claude/GPT API）
- [ ] 前端实现问答面板 UI
- [ ] 上线测试，收集反馈

### Phase 2（3-4周）：质量优化

- [ ] 人工为历史案例补充「解决方案」字段
- [ ] 实现自动标签提取（调 LLM 批量处理历史数据）
- [ ] 优化 Prompt，提升回复质量
- [ ] 添加「这个回答有用/没用」反馈按钮，收集评价数据

### Phase 3（后续）：向量化升级（可选）

- [ ] 引入 Embedding API（OpenAI / Cohere）
- [ ] 接入轻量向量存储（如 Chroma / Supabase pgvector）
- [ ] 替换关键词检索为语义检索，提升召回质量
